---
title: "Sharing safely"
description: "Understand Tunelito's review-key access model and tunnel exposure."
---

Tunelito is built for temporary collaboration, not permanent hosting.

## Review keys

Shared sessions include a generated `tunelito_key` in printed URLs by default:

```text
https://example.trycloudflare.com/?tunelito_key=...
```

The key is bearer access. Anyone with the full URL can view the page and leave comments.

The first valid request sets a short-lived HTTP-only cookie so page assets and WebSocket sync keep working.

## Owner labels

`--owner <name>` adds a separate owner key to the printed `Local:` URL. Comments from that owner session are marked as owner-authored, while public tunnel URLs still only carry the review key.

The owner key is a collaboration label, not stronger authentication. Anyone with the full owner URL can comment as the owner, so do not share the `Local:` owner URL when owner-only agent instructions matter.

## Public tunnels

When tunneling is enabled, Tunelito exposes your local server through Cloudflare Tunnel. Use this only for material you are comfortable sharing with anyone who has the full link.

## Ephemeral live sessions

`--live` keeps comments in memory only and does not write a comments file. It changes persistence, not exposure: the session is still served over the same tunnel and review key, so "nothing saved" is not "nothing exposed." The same bearer-link access model still applies: anyone with the full URL can view the page and participate while the local server is running. For a sensitive page, combine `--live` with `--no-tunnel`.

WebRTC data channels are encrypted by the browser when peer-to-peer connectivity is available. The WebSocket connection remains active for signaling, reloads, and fallback relay. Tunelito does not configure third-party STUN or TURN servers.

## Local agent worker

The agent worker is explicit opt-in. When `--agent` is enabled, reviewer comments become input to a local CLI that can edit files in the served folder. Use it for trusted review sessions, add `--agent-trigger "@agent"` so only marked comments are actionable, or combine owner identity with `--agent-policy owner-or-mention --agent-trigger "@agent"` for owner-led sessions. Tunelito blocks the agent state file, temp file, and derived log file from static serving even if a custom `--agent-state` path is inside the served folder.

`--agent-session` has the same trust boundary but does not spawn a child process. The server process watches comments, claims actionable feedback for the current agent session, prints prompts with `tunelito inbox record --claim ...` commands, shows the current work on browser comment cards and through `tunelito inbox status`, and writes `.tunelito/session.json`. Reviewer comments still become instructions to the current local agent session, so use the same owner or mention policies for shared calls.

## Disable public tunnels

```bash
tunelito ./page.html --no-tunnel
```

## Disable review-key auth

```bash
tunelito ./page.html --no-auth
```

`--no-auth` only removes the review-key gate; it does **not** disable the tunnel. A tunneled session started with `--no-auth` is a public, unauthenticated URL that anyone who finds it can open and edit. If you want no key, add `--no-tunnel` so the session stays local. Use `--no-auth` only for trusted local networks or throwaway demos.
