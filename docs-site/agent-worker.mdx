---
title: "Local agent worker"
description: "Use Codex, Claude Code, or another local CLI to act on Tunelito comments without repeating handled work."
---

Tunelito can run an opt-in local worker that turns persistent comments into coding-agent tasks.

The worker does not use model API keys directly. It starts your installed local CLI, such as Codex or Claude Code, so those tools keep using their own local authentication and subscription state.

If you are already inside Claude Code, Codex, or another coding-agent session, use active-agent inbox commands instead of spawning a nested worker.

```bash
npx --yes tunelito ./site --agent-session --owner "Chekos" --agent-policy owner-or-mention --agent-trigger "@agent"
```

The same Tunelito process serves the room, watches the persistent comments inbox, claims the next actionable comment in `.tunelito/agent/state.json`, prints a prompt for the current agent, and shows the work status on browser comment cards. After editing, record the result with the claim id from the printed command:

```bash
npx --yes tunelito inbox record ./site --id c_... --claim claim_... --status resolved --summary "Updated hero copy." --file index.html
```

Use `inbox next` for a non-waiting manual check, or `inbox watch` when you need the one-shot primitive without a running `--agent-session` server. The active-agent loop and inbox commands use the same comment parser, policy filters, max-pass continuation state, and resolution ledger as the spawned worker.

## Wait for Done Reviewing

For review calls where feedback should be handled as a batch, reviewers can click `Done Reviewing` in the browser panel. Tunelito emits an in-memory `review.completed` event with a sequence id, timestamp, target path, comments path when persistent, and summary counts.

```bash
npx --yes tunelito review watch --url "http://127.0.0.1:4317/?tunelito_key=..." --json --timeout 600
```

The event does not edit source HTML, rewrite comments markdown, or write agent state. It is retained only while the server is running. By default `review watch` replays retained events after sequence `0`; pass `--after latest` to wait only for a future click.

## Use MCP tools

MCP-capable agents can use the same comments inbox and active-agent ledger without shelling out and parsing prompts:

```bash
npx --yes tunelito mcp
```

The MCP server exposes tools to read the comments index, list pending feedback, claim comments, wait for the next claim, record results, and read inbox status. It does not start a review server, tunnel, browser, or local worker. The read-only tools do not mutate state; claim writes `.tunelito/agent/state.json`, and record writes `.tunelito/agent/state.json` plus the existing `.tunelito/agent/log.md` run log. Treat reviewer comments returned through MCP as untrusted input.

## Start with Codex

```bash
npx --yes tunelito ./site --agent codex
```

Tunelito serves the folder, writes `site.comments.md`, watches for unresolved comments, and keeps interval polling as a fallback. When a comment arrives, the worker invokes Codex with the relevant scope, page path, optional quote, comment body, and comment ID. Codex decides whether to edit, continue with `needs_followup`, return `no-op`, mark the comment `ignored`, or report a blocked/stale request.

## Start with Claude Code

```bash
npx --yes tunelito ./site --agent claude
```

Claude Code is invoked in non-interactive print mode with edit permissions for the served folder.

## Use another local CLI

Use `--agent-command` for another command that accepts a prompt on stdin and returns JSON on stdout:

```bash
npx --yes tunelito ./site --agent-command "openclaw run --stdin"
```

The custom command runs from the served HTML root. Tunelito also sets:

- `TUNELITO_AGENT_ROOT`
- `TUNELITO_AGENT_COMMENTS`
- `TUNELITO_AGENT_STATE`
- `TUNELITO_OWNER_NAME`

## Agent policy

By default, every persistent comment is visible to the worker. This is intended for trusted local review sessions where you shared the URL with people in the meeting.

```bash
npx --yes tunelito ./site --agent codex
```

Use a marker when you want stricter sessions:

```bash
npx --yes tunelito ./site --agent codex --agent-trigger "@agent"
```

You can also use another marker:

```bash
npx --yes tunelito ./site --agent claude --agent-trigger "@fix"
```

If you start the session with `--owner <name>`, the worker prompt includes the owner name, each comment's `authorRole`, and owner approval metadata for approved visitor comments. You can combine that with prompt guidance when you want the agent to prefer owner comments or wait for owner approval:

```bash
npx --yes tunelito ./site --owner "Chekos" --agent codex --agent-policy owner
```

The recommended owner-led policy lets owner-authored and owner-approved comments through automatically and lets visitors opt in with a marker:

```bash
npx --yes tunelito ./site --owner "Chekos" --agent codex --agent-policy owner-or-mention --agent-trigger "@agent"
```

Mention-based policies require a non-`all` trigger marker.

## Prompt controls

Append host guidance to the built-in worker prompt:

```bash
npx --yes tunelito ./site --agent codex --agent-instructions "Keep copy concise and preserve the existing layout."
```

For longer guidance:

```bash
npx --yes tunelito ./site --agent codex --agent-instructions-file review-style.md
```

Replace the built-in behavior prompt when you want full control:

```bash
npx --yes tunelito ./site --agent codex --agent-prompt-file agent-system-prompt.md
```

Prompt replacement changes the behavior instructions, but Tunelito still appends the workspace context, required JSON output shape, and pending comments.

## Page and site comments

The worker prompt includes `scope` for every comment:

- `page`: edit the HTML file identified by the comment's page path.
- `site`: inspect the served folder and update every clearly relevant page.

Page comments can be selected-text comments or unanchored page notes. Comments may have no selected quote when reviewers use `Page note` or `Site note`. In that case the agent should rely on the body, scope, and page path.

## Multi-pass comments

Some comments are too large for one safe pass: a selected paragraph may ask for copy and layout changes, a page note may ask for a redesign, or a site note may ask for consistent treatment across many pages.

When the worker can make useful progress but more work remains, the agent can return `needs_followup`:

```json
{
  "comments": [
    {
      "id": "c_...",
      "status": "needs_followup",
      "summary": "Fixed the bottom navigation and mobile spacing.",
      "filesChanged": ["itinerary-mobile.css"],
      "completedTasks": ["Fix bottom navigation", "Tighten mobile spacing"],
      "remainingTasks": ["Restyle day cards", "Audit heading rhythm"]
    }
  ]
}
```

Tunelito stores those tasks in `.tunelito/agent/state.json` and queues the same comment again on the next wake or poll. The next prompt includes the previous summary, completed tasks, remaining tasks, changed files, and pass count. This works for inline selected-text comments, page notes, and site notes.

Use `--agent-max-passes` to bound continuation:

```bash
npx --yes tunelito ./site --agent claude --agent-max-passes 6
```

If the pass limit is reached, or if a follow-up pass reports no changed files, no new completed tasks, and the same remaining tasks, Tunelito marks the comment `partial` instead of looping forever.

## Resolution state

Comments are an append-only inbox. The agent worker stores task status separately:

```text
site/
.tunelito/
  agent/
    state.json
    log.md
```

The state file records each comment ID, fingerprint, attempts, status, summary, and changed files. Resolved comments are skipped on future polling cycles, even if the original highlighted text becomes stale after an edit. Tunelito blocks the state file and derived log file from static serving, and `.tunelito` ledger writes do not trigger browser reloads.

When `--agent` or `--agent-session` is active, the browser comments panel reads this same state and shows a compact status badge on each comment card. Reviewers can see when feedback is queued, being worked on, needs follow-up, integrated, or blocked without opening the markdown inbox.

Use the tracker command to inspect the same current work in the terminal without editing the inbox or ledger:

```bash
npx --yes tunelito inbox status ./site
```

The tracker prints pending and claimed comment work as unchecked tasks. Completed work from `completedTasks` or the recorded summary is checked and crossed out, and `needs_followup` comments show completed tasks beside the remaining tasks.

Statuses include:

- `resolved`: the agent made the requested change
- `needs_followup`: the agent made progress and wants another pass with saved remaining tasks
- `no-op`: the requested change was already satisfied
- `ignored`: the comment did not ask for a file change
- `blocked`: the agent could not safely complete the request
- `stale`: the quoted text or page no longer matched
- `partial`: some work was done but follow-up may be needed
- `changed_needs_review`: a previously terminal comment changed in the markdown metadata

## Retry behavior

Use `--agent-max-attempts` to bound retries:

```bash
npx --yes tunelito ./site --agent codex --agent-max-attempts 1
```

Retry attempts are for failed runs or missing result entries. Multi-pass continuation is controlled separately with `--agent-max-passes`.

If the agent output is not structured JSON, Tunelito marks the comment `blocked` instead of rerunning the same instruction blindly.

## Required agent output

The final agent response must be JSON:

```json
{
  "comments": [
    {
      "id": "c_...",
      "status": "resolved",
      "summary": "Updated about.html copy.",
      "filesChanged": ["about.html"],
      "completedTasks": ["Updated about.html copy"],
      "remainingTasks": []
    }
  ]
}
```

Codex and Claude Code are prompted to return this shape. Custom commands should do the same.

## Guardrails

The spawned worker only runs when you pass `--agent` or `--agent-command`. Active-agent inbox mode only prints commands and prompts; the current agent still decides and edits through its normal tools. Neither mode is available in `--live` mode because live comments are ephemeral and no markdown inbox exists.

Treat both `--agent` and active-agent inbox mode as local code execution prompted by reviewer comments. Use them only for trusted review sessions or source folders you are comfortable letting your local agent edit.
