title: "How it works"
description: "Understand the Tunelito local server, injected client, live sync, and persistence modes."
---

Tunelito is local-first. It does not upload your source HTML to a hosted backend.

## Runtime pieces

The CLI starts a local server for one HTML file. The server reads that file from disk, injects the Tunelito browser client into the served response, and serves sibling assets from the same directory.

The browser client handles:

- text selection
- comment composition
- comment sidebar
- live sync over WebSocket, plus WebRTC data channels in `--live`
- highlight rendering when the browser supports it
- reload notices when the file changes

The server handles:

- review-key access control
- WebSocket upgrades
- WebRTC signaling and fallback relay events in `--live`
- comment validation
- Markdown persistence, unless `--live` is enabled
- file watching
- optional Cloudflare Tunnel startup

## The HTML file is not modified

The injection happens only in the HTTP response. Your source HTML file remains untouched.

## Persistence

Comments are written to Markdown beside the page by default:

```text
page.html
page.comments.md
```

The Markdown remains readable if Tunelito is never opened again. Hidden metadata inside the file lets Tunelito restore live comments after a restart.

## Live mode

`--live` stores comments in memory only. The browser client shares comments, cursors, and selection highlights over WebRTC data channels when possible. The WebSocket connection remains the authenticated control channel for room membership, WebRTC signaling, reload notices, and fallback relay when peer-to-peer connectivity is unavailable.
