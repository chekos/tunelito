title: "How it works"
description: "Understand the Tunelito local server, injected client, live sync, and persistence modes."
---

Tunelito is local-first. It does not upload your source HTML to a hosted backend.

## Runtime pieces

The CLI starts a local server for one HTML file or a folder of HTML files. The server reads HTML from disk, injects the Tunelito browser client into served HTML responses, and serves non-hidden assets from the selected file directory or folder root.

The browser client handles:

- text selection
- comment composition
- comment sidebar
- live sync over WebSocket, plus WebRTC data channels in `--live`
- highlight rendering when the browser supports it
- reload notices when the file changes

The server handles:

- review-key access control
- WebSocket upgrades
- WebRTC signaling and fallback relay events in `--live`
- comment validation
- Markdown persistence, unless `--live` is enabled
- file watching
- page-specific comment streams when serving a folder
- optional Cloudflare Tunnel startup

## Source HTML is not modified

The injection happens only in the HTTP response. Your source HTML files remain untouched.

## Persistence

Comments are written to Markdown beside the page by default:

```text
page.html
page.comments.md
```

For folder targets, the default inbox is beside the folder:

```text
site/
site.comments.md
```

The Markdown remains readable if Tunelito is never opened again. Hidden metadata inside the file lets Tunelito restore live comments after a restart. Folder comments include visible page paths so a person or coding agent can map feedback back to the right HTML file.

## Live mode

`--live` stores comments in memory only. The browser client shares comments, cursors, and selection highlights over WebRTC data channels when possible. The WebSocket connection remains the authenticated control channel for room membership, WebRTC signaling, reload notices, and fallback relay when peer-to-peer connectivity is unavailable.
