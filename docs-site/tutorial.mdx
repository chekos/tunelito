---
title: "Tutorial: folder review with agent follow-up"
description: "Serve a folder of HTML files, collect page and site comments, and have a local coding agent edit the matching source files."
---

This walkthrough uses Tunelito as a local review server for a folder-backed mini-site. Reviewers leave selected-text comments, page notes, or site-wide notes in the browser. Tunelito writes one Markdown inbox, and an opt-in local agent worker can call Codex, Claude Code, or another CLI to edit the matching HTML files.

## 1. Create a small HTML folder

Use an existing folder, or create a quick test site:

```bash
mkdir -p review-site

cat > review-site/index.html <<'HTML'
<!doctype html>
<html>
  <body>
    <main>
      <h1>Launch draft</h1>
      <p>This paragraph is ready for review.</p>
      <a href="/about.html">About</a>
    </main>
  </body>
</html>
HTML

cat > review-site/about.html <<'HTML'
<!doctype html>
<html>
  <body>
    <main>
      <h1>About this project</h1>
      <p>Explain the project here.</p>
    </main>
  </body>
</html>
HTML
```

## 2. Start Tunelito with an agent worker

For local-only review:

```bash
npx --yes tunelito ./review-site --agent codex --agent-max-passes 5 --no-tunnel --open
```

For a shareable public tunnel:

```bash
npx --yes tunelito ./review-site --agent codex
```

Use `--agent claude` for Claude Code. Use `--agent-command "<your command>"` for another local CLI that accepts a prompt on stdin and returns JSON. `--agent-max-passes` lets one large comment continue across several bounded passes.

If you are already inside an agent session, use active-agent inbox mode instead of spawning Codex or Claude Code again:

```bash
npx --yes tunelito ./review-site --agent-session --agent-policy owner-or-mention --agent-trigger "@agent" --no-tunnel --open
```

Tunelito prints a local URL, a comments file, and, when tunneling works, a public URL:

```text
Tunelito is running
Local:   http://127.0.0.1:4317/?tunelito_key=...
Comments: /path/to/review-site.comments.md
Agent:   codex; checks all comments every 120s; max 5 passes; state /path/to/review-site/.tunelito/agent/state.json
Access:  review key required by the printed URLs
Public:  https://example.trycloudflare.com/?tunelito_key=...
```

Share the `Public:` URL with reviewers. Keep the terminal process running while the review is active.

If the folder should behave like an iPhone Home Screen app, add site-level web app metadata before sharing. See the [iPhone Home Screen guide](/iphone-home-screen).

## 3. Leave comments

Open the served page and try three comment types:

1. Select text and click `Comment`. This creates an anchored page comment and highlights the selected text when possible.
2. Open the comments panel and click `Page note`. This creates an unanchored note for the current page.
3. Click `Site note`. This creates a note that appears on every page in the folder review.

On another page, such as `/about.html`, comments are still written to the same Markdown file. Each comment includes a scope and page path.

The comments file will look like this:

```markdown
# Tunelito comments for `review-site`

## Jane at 2026-05-26 21:00:00 UTC

> Explain the project here.

Make this sentence more specific.

_Context: scope: `page` · page: `/about.html` · path: `body > main > p` · text offset: 18 · id: `c_...`_

## Jane at 2026-05-26 21:03:00 UTC

_Site note (no selected text)._

Use the same heading rhythm on every page.

_Context: scope: `site` · page: `/about.html` · id: `c_...`_
```

No marker is required by default. If you start Tunelito with `--agent-trigger "@agent"`, include `@agent` in comments that should be handled.

## 4. Let the worker handle comments

The worker watches the comments file, sends unresolved comment IDs to the selected local CLI, keeps interval polling as a fallback, and records status in:

```text
review-site/
.tunelito/
  agent/
    state.json
    log.md
```

When the agent saves HTML changes, Tunelito asks connected browsers to reload. If a reviewer has an open comment composer, the reload is queued until the draft is submitted or closed so unsubmitted text is not lost. Page-scoped comments point the agent at one page. Site-scoped comments tell it to inspect the folder and update every clearly relevant page.

If a selected-text comment, page note, or site note needs more than one pass, the agent can return `needs_followup` with completed and remaining tasks. Tunelito stores that continuation in the state file and sends the same comment again on the next wake or fallback poll. If the agent reaches `--agent-max-passes` or stops making progress, Tunelito marks the comment `partial`.

Resolved comments are skipped on future worker passes and show as integrated in the browser comment panel. If editing the HTML makes the original highlighted quote stale, the comment still stays resolved in the ledger and will not be applied again.

When you use `--agent-session`, the Tunelito server process is the active-agent loop. It waits for actionable comments, claims one for the current agent session, updates the browser comment-card status, prints the prompt, and pauses further claims until that claim is recorded or expires. After editing, record the result with the claim id from the printed command:

```bash
npx --yes tunelito inbox record ./review-site --id c_... --claim claim_... --status resolved --summary "Updated the about page copy." --file about.html
```

## 5. Manual fallback

If you do not want Tunelito to run the worker, start without `--agent`:

```bash
npx --yes tunelito ./review-site
```

Then ask a coding agent to inspect the Markdown inbox:

```text
Monitor review-site.comments.md every 2 minutes. For each new actionable comment, use `scope: page` comments to edit the matching HTML file and `scope: site` comments to update all clearly relevant pages. If one comment needs multiple passes, keep a completed/remaining task list so the next pass continues instead of repeating work. After each change, report the comment id and what changed.
```

For one manual pass instead of polling:

```text
Read review-site.comments.md, address every actionable unresolved page or site comment in the matching HTML files, then summarize comment ids and edits.
```

## 6. Stop the review

Press `Ctrl+C` in the Tunelito terminal. Persistent comments remain in `review-site.comments.md`.

Use `--live` only for ephemeral in-meeting collaboration. In live mode, comments are not written to Markdown, so there is no file for an agent to monitor.
